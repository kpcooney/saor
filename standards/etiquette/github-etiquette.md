# GitHub Etiquette

Conventions for how agents and humans interact on GitHub — issues, PRs, and comments. The goal is signal-rich, low-noise communication that makes the repository useful as a historical record.

## Issues

**Keep issues focused.** One issue = one outcome. If you discover a second problem while working on an issue, open a new issue rather than expanding the scope of the current one.

**Use task lists for multi-step work.** If an issue has multiple acceptance criteria, format them as GitHub checkboxes (`- [ ] item`). Check them off as they're completed — it makes progress visible without a status update comment.

**Reference related work.** If an issue is related to an ADR, link it. If it blocks or is blocked by another issue, use GitHub's "linked issues" feature or mention the issue number in the body (`Blocked by #12`, `Depends on #18`).

**Do not use issue comments to provide status updates** unless there is a decision or blocker to communicate. "Working on it" is noise. Update the checklist items and let the PR speak for itself.

## Pull Requests

**One PR per issue.** A PR should close exactly one issue. If your work touches multiple concerns, split it — it makes review easier and the history cleaner.

**Link the issue in the PR body.** Use `Closes #NNN` in the PR body (not the title). GitHub will automatically close the issue when the PR is merged.

**Do not merge your own PRs.** Kevin merges. This applies to all agents. Even if you have the repository permission to merge, do not use it on your own PRs.

**Respond to every review comment.** If you addressed a comment, say how. If you disagree, explain why. Do not silently push a commit that may or may not have addressed a comment — make it explicit.

**Do not force-push a branch under review.** Push additional commits in response to review feedback. This preserves the review conversation's context. Squash only after approval and before merge, and only if the reviewer requests it.

## Comments

**Be specific.** A comment that says "I changed the approach here" is less useful than "I changed from string concatenation to parameterized queries here because the previous approach was vulnerable to SQL injection — see the security checklist."

**Reference code with line anchors.** On GitHub, you can link to a specific line in a file. Use these when discussing specific code in an issue or PR comment — it removes ambiguity.

**Agents must identify themselves.** When an agent posts a comment or opens an issue or PR, the body must include a note identifying the agent:

```
---
*Generated by the Code Agent (agent:code:proj-100:sprint-42) as part of [PROJ-167](link).*
```

This creates the traceability chain between GitHub activity and the audit log.

## Labels

Apply labels consistently. Every issue should have exactly one type label and one scope label:

**Type labels:**
- `feat` — new feature or capability
- `fix` — bug fix
- `chore` — tooling, configuration, scaffolding
- `docs` — documentation changes only
- `refactor` — code restructuring, no behavior change
- `test` — adding or fixing tests

**Scope labels:**
- `tauri` — Rust backend
- `frontend` — Svelte frontend
- `agents` — TypeScript agent layer
- `memory` — memory store
- `audit` — audit store
- `mcp` — MCP server definitions
- `standards` — standards files

## Requesting Review

Request review when the PR is marked ready for review (not draft). Do not ping reviewers via @mention in every comment — one mention when the PR is ready is sufficient. If a PR has been open for more than two working days without a review, a single follow-up is appropriate.

## Keeping Conversations Focused

GitHub issues and PRs are not chat. Each comment should add information — a decision, a question, a clarification of requirements, or a response to a review comment. Keep side discussions in separate issues. Keep the PR focused on the code being reviewed.
